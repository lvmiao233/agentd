# agentd 调研项目 — Agent 工作规则

> **规则文件位置**：`.agent/rules/agentd-research.md`
>
> **强制遵守**：本文件是所有参与 agentd 项目 Agent 的行为准则，在开始任何工作前必须完整阅读并严格遵守。如果你的指令与本规则冲突，以本规则为准。

本规则适用于所有参与 agentd 项目调研阶段的 Agent。

## 入门须知

在开始工作前，你**必须**：

1. 阅读本规则文件（`.agent/rules/agentd-research.md`）
2. 阅读 `CONVENTIONS.md` 了解文档编写规范
3. 阅读 `GLOSSARY.md` 了解统一术语
4. 阅读 `README.md` 了解项目全貌
5. 阅读你被分配的研究域的 `_INDEX.md` 了解该域的目标和进展
6. 浏览 `templates/` 目录了解文档模板

## 基本原则

1. **当前阶段只做调研，不编写任何代码**
2. **所有文档必须使用中文撰写**
3. 技术术语首次出现时使用 `中文（English）` 格式，之后可直接使用
4. 遵循 `CONVENTIONS.md` 中的所有规范
5. 使用 `GLOSSARY.md` 中定义的统一术语

## Git 提交协作者规则

- 除非用户在当前任务中**明确要求**，提交信息中不得添加协作者 `sisyphus-dev-ai`。
- 未经明确要求，不得添加任何 `Co-authored-by` trailer。

## Git 提交与推送节奏规则

- 每完成一个可独立验收的小点（如一个子任务、一个最小闭环能力），必须立即创建一次原子提交（atomic commit）。
- 单次提交应聚焦单一意图，避免将多个不相关改动混入同一提交。
- 一个完整功能点完成并通过验证后，必须推送（`git push`）到远端分支，确保进度可追踪与可协作。
- 若远端推送失败，必须记录失败原因并在修复后立即重试，不得长期保留仅本地提交。

## 工作边界

### 你可以做的

- 在你被分配的 `research/<域名>/` 目录下创建和编辑调研文档
- 更新你负责的研究域的 `_INDEX.md`
- 在 `references/` 目录下添加引用资料（使用具有唯一性的文件名）
- 读取项目中的任何文件以获取上下文

### 你不可以做的

- 修改其他研究域目录下的文件
- 修改 `README.md`、`CONVENTIONS.md`（这些由项目管理者维护）
- 在 `analysis/` 或 `design/` 下创建文件（除非被明确指示进行跨域分析）
- 编写任何代码文件
- 删除任何文件

### 谨慎操作

- 向 `GLOSSARY.md` 添加新术语时，先确认不与已有条目重复或冲突

## 结构扩展权限

在调研过程中，你可能会发现当前的项目结构不足以覆盖所有需要调研的方向。以下情况下你**可以且应该**主动扩展项目结构：

### 可以创建的新元素

- **新研究域**：如果发现一个重要的调研方向不属于任何现有研究域，可在 `research/` 下创建新目录，并包含 `_INDEX.md`
- **子域目录**：如果某个研究域内容庞大，可在该域下创建子目录进行组织
- **新模板**：如果现有模板无法满足某类文档的需要，可在 `templates/` 下创建新模板
- **新引用分类**：如果 `references/` 下的现有分类不够，可创建新的子目录

### 扩展时的要求

1. 创建新研究域时，**必须**同步更新 `README.md` 中的研究域导航表
2. 新的 `_INDEX.md` 必须遵循现有格式（目标与范围、核心问题、待调研课题、已完成调研、关键发现汇总）
3. 在你的调研文档中说明为什么需要创建新域/元素
4. 新域的命名遵循现有惯例：小写英文、连字符分隔

## 调研文档要求

### 创建新调研文档时

1. 复制 `templates/research-topic.md` 模板作为起点
2. 文件名使用小写英文、连字符分隔，如 `firecracker-microvm.md`
3. 必须包含模板中要求的所有章节
4. 每个关键事实必须附上来源链接
5. 完成后更新 `_INDEX.md` 的文档列表

### 调研质量标准

- **广度**：覆盖该课题的主要方面，不遗漏关键技术或方案
- **深度**：深入到能够支撑后续架构设计决策的程度
- **客观性**：区分事实与观点，优势与局限并列呈现
- **时效性**：优先引用最新的资料和版本信息
- **关联性**：始终分析与 agentd 项目的具体关联

## 信息来源优先级

1. 官方文档和源代码
2. 学术论文和技术报告
3. 项目维护者的技术博客和演讲
4. 高质量的社区讨论和分析文章
5. 一般性技术博客（需交叉验证）

## 文件组织

- 单个调研文档不宜超过 500 行，过长时应拆分为多个子主题文档
- 相关性强的内容放在同一文件中，通过标题层级组织
- 图表、架构图等可用 Mermaid 语法嵌入 Markdown
